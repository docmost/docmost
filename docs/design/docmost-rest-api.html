Docmost · REST API 分析與設計

Docmost REST API:商業版有什麼、為什麼存在、開源版怎麼設計

拆解官方企業版 API 的實際功能與授權模型,說明它為何被設計、又為何鎖在 Enterprise;接著給出一套可自建的開源版 REST API 設計 —— 具體端點、技術難點與應用範圍。

2026 年 6 月 · 依據 Docmost 官方 API 文件(Scalar / OAS 3.1.0, v1.0.0)、原始碼結構與社群實作整理

一、商業版 Docmost REST API 有什麼功能

官方 API 是 Enterprise 專屬功能(需有效企業授權)。它是一套以 OpenAPI 3.1 描述的 RESTful 介面,有幾個一致慣例:

  • 認證:傳 Authorization: Bearer <token>。token 是個人 API key,綁定建立者、繼承該使用者的權限;可設到期(30/60/90/365 天、自訂、永不),可命名、改名、撤銷;管理員可在 workspace 層級檢視與管理所有人的 key。
  • 呼叫慣例:除少數標註外,端點一律用 POST(偏 RPC-over-POST 風格);列表用 cursor 分頁(limit 1–100),搜尋回 data.items[]。

白話講:它幾乎把 Docmost UI 能做的事都開成端點 —— 建/改/刪頁面與空間、上傳附件、搜尋、管理群組與權限、匯入匯出、佈建使用者與 SSO。注意頁面「內容」的寫入在底層仍與即時協作(Yjs)綁定,這點在開源版設計時是最大關卡。

二、為什麼要設計這個 REST API

分兩個層次:為什麼需要一個程式化 API,以及Docmost 為什麼把它鎖在 Enterprise。

A. 為什麼需要程式化 API(它解決什麼)

內容自動化:讓外部系統把文件「寫進」wiki —— CI/CD 自動產生 API 文件、把監控告警的 runbook、事故 postmortem 自動歸檔成頁面。
批次遷移:從 Confluence / Notion 大量搬入,保留階層與附件,人工點不完。
AI / Agent 整合:MCP 服務背後就是這套 API。讓 agent 讀知識、寫回整理結果 —— 正是前面 Agentic 知識層要的「投影前台」。
佈建與治理:自動建立空間、佈建使用者群組、套權限(類 SCIM),配合 SSO 做企業身分整合。
備份與合規:定期匯出做備份、稽核、離線保存(對 ITAR/GDPR 類合規場景重要)。

B. 為什麼鎖在 Enterprise

這是商業模式的選擇,不是技術限制。可自架、不限人數是社群版的賣點;而「程式化整合 + 治理」恰好是企業最需要、也最願意付費的能力:

  • API 本身就是整合的價值點 —— 企業要把 wiki 接進既有系統,這條整合線就是營利線。
  • API key 的治理(scoping、到期、稽核「last used」、管理員集中管理、撤銷)正是企業合規要的,把它做成付費功能順理成章。
  • 連帶的 SSO、MFA、權限審批 也都歸在企業層,API 與它們是同一套治理故事。
換句話說:免費版讓你「自己用」,企業版讓你「接進組織的系統與流程」。API 是後者的核心。所以若你只是要程式化操作自架的社群版,自建一套 API 在技術上完全可行(核心是 AGPL 開源)。

三、如何設計開源版的 Docmost REST API

四個設計原則

  • 不碰 ee 程式碼:核心是 AGPL,可自由建構;但 packages/ee 通常另有商業授權,自己重寫對等模組,不要移植。AGPL 的網路 copyleft 也要注意。
  • 讀寫分離:讀走 PostgreSQL 或內部端點;寫「頁面內容」必須經 Hocuspocus / Yjs(下詳),不能直接改 DB。
  • REST-native:設計成資源導向(而非官方的 RPC-over-POST),自由度高、好用、好快取。
  • 權限繼承:沿用 Docmost 既有的 space role 與 CASL 權限檢查,用服務帳號身分執行,避免另造一套權限系統。

建議端點規格(MVP → 完整)

MVP 先打通內容流;完整 補組織與搬移;加值 是官方沒有、但知識層很需要的。

關鍵技術設計

最硬的一塊:頁面內容寫入。內容存成 Yjs CRDT 二進位,直接寫 Postgres 會破壞協作合併。正確做法是當一個 Yjs 客戶端,連到 :3001 的 Hocuspocus,用 @hocuspocus/transformer 的 TiptapTransformer 把 ProseMirror JSON 轉成 Y.Doc 再套用,由 Hocuspocus 負責持久化。transformer 要載入 packages/editor-ext 同一組擴充,結構才會被認得。
// 寫入路徑骨架（sidecar 內）
const ydoc = TiptapTransformer.toYdoc(prosemirrorJSON, "default", extensions);
const provider = new HocuspocusProvider({
  url: "ws://localhost:3001",
  name: `page.${pageId}`,      // Docmost 文件房間命名
  document: ydoc,
  token: serviceAccountJWT,
});
// Hocuspocus 收到後 merge → 寫回 Postgres，內容與協作狀態皆正確
  • 讀路徑:直接查 pages 表(ProseMirror JSON、階層、空間),最快;附件走既有 storage 抽象(本機 / S3)。
  • API key:雜湊後存表,記 scope 與到期;比對時 hash 比對,記錄 last_used 供稽核。
  • 權限:每個請求解析出對應使用者身分 → 套用 space CASL 檢查,不另造權限模型。
  • 慣例:cursor 分頁(limit 1–100)、一致錯誤模型(code/message/details)、基本 rate limit。

架構選型:sidecar vs fork

做法怎麼做取捨
Sidecar(推薦)獨立 FastAPI/Express 服務:讀 DB、寫經 Hocuspocus不動 Docmost 本體、升級不衝突;需自行對齊 schema
NestJS 模組在 fork 內加一個對等 API module 掛進主程式整合最乾淨、共用權限;受 AGPL copyleft、升級需 rebase

四、應用範圍

  • Agentic 知識層的前台投影:前面設計的 project-to-docmost skill 就用這套 API,把編譯好的 artifact 與查詢結果寫成 Docmost 頁,並用回溯連結指向來源版本。
  • 文件 CI/CD:程式碼倉庫變更 → 自動產生 / 更新 API 文件、架構決策紀錄(ADR)、runbook。
  • 外部系統內容同步:工單、監控、ChatOps 把事件與 postmortem 自動歸檔。
  • 遷移與備份:大量匯入舊系統內容、定期匯出做合規備份。
  • 佈建:依組織結構自動建空間、群組、套權限。
  • 加值:webhooks —— 官方沒有,但知識層需要監聽頁面變動來觸發 re-cognify / 重編,值得在 OSS 版優先補上。
務實建議:先做 MVP 七條(key 自管 + pages CRUD + search + 附件 + import),把「能程式化讀寫頁面」打通;唯一的硬骨頭是 Yjs 內容寫入,用官方 transformer 正確解掉。之後再補組織/搬移,並把 webhooks 當成你比商業版多出來的價值。
